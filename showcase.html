Process Design · Horses for Courses

UAT Simplification: 20 Excel tabs to a Guided Portal

The problem

Fifty users completing UAT across twenty Excel tabs. Everyone had edit access. No data validation. Testers had to read instructions before adding a row. Confidence with Excel varied. Leaders were openly concerned about usability.

What I built

A guided testing solution using PowerPoint as the front end and Jira Service Management on the back end. Not the tool I would normally reach for, but the one users already knew, which meant near zero training overhead. Built in a few days.

Why this approach

An AI-native or purpose-built solution would have been overkill for the timeframe and audience. Familiar tooling, plus a proper back-end that gave the team real oversight, delivered a much better experience than the twenty tabs and a much lower risk profile than a rushed build.

Personal Project · Full-Stack SaaS

Multi-Environment SaaS Application

The problem

A personal project, built to answer a specific question. Can someone with a business analyst background and one more technical collaborator ship a proper multi-environment SaaS product, with the deployment discipline you would expect from something built for actual use, without going through a full engineering team?

What I built

A full SaaS application on [add stack, e.g. Supabase, GitHub and Vercel]. The point of the project was not the feature set. It was the practices underneath it. Separate development and production databases from day one, not two schemas in one project. Row-level security enabled before any user existed, because it is far harder to retrofit than to start with. Schema changes committed as migration files, not made by hand in a dashboard. Branch protection requiring a review before anything merges to production. Secrets in encrypted environment stores, never in the source code.

Why this matters

Most of the "AI-assisted build" conversation stops at the static HTML file. This project was the next step. The interesting problem is not how fast you can generate a working app. It is how you keep the fast, ungoverned moment of describing what you want separate from the moment it reaches real users or real data. Staging environments, required review and automated checks are all different implementations of that one idea. This was the smallest version of that principle applied to a real build, and the answer to what a small team actually needs to build safely at scale.
